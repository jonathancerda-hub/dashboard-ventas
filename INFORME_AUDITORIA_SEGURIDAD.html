Hallazgos Detallados de Seguridad

1. CSP con 'unsafe-inline' y 'unsafe-eval'
Media
Categoría OWASP A05:2021 - Security Misconfiguration
Control ISO 27001 A.14.2.5 - Principios de ingeniería segura
Archivo Afectado app.py (líneas 102-113)

📋 Descripción

La Content Security Policy permite scripts y estilos inline mediante 'unsafe-inline' y 'unsafe-eval'. Esto reduce significativamente la efectividad de CSP como defensa contra XSS, permitiendo ejecución de código JavaScript arbitrario inyectado.

ISO/IEC 27001:2022 - Control A.14.2.5: Los principios de ingeniería de sistemas seguros requieren implementar controles que minimicen la superficie de ataque. El uso de 'unsafe-inline' amplía innecesariamente esta superficie.

💻 Código Vulnerable

csp_directives = [ "default-src 'self'", "script-src 'self' 'unsafe-inline' 'unsafe-eval' https://accounts.google.com", "style-src 'self' 'unsafe-inline' https://fonts.googleapis.com", # ⚠️ VULNERABLE: unsafe-inline y unsafe-eval permiten XSS ] response.headers['Content-Security-Policy'] = "; ".join(csp_directives)

✅ Remediación Recomendada

Implementar nonces dinámicos para scripts y estilos específicos:

import secrets @app.after_request def add_security_headers(response): nonce = secrets.token_urlsafe(16) g.csp_nonce = nonce # Disponible en templates csp_directives = [ "default-src 'self'", f"script-src 'self' 'nonce-{nonce}' https://accounts.google.com", # ✅ Remover unsafe-inline y unsafe-eval f"style-src 'self' 'nonce-{nonce}' https://fonts.googleapis.com", "object-src 'none'", "base-uri 'self'", ] response.headers['Content-Security-Policy'] = "; ".join(csp_directives) return response # En templates Jinja2: <script nonce="{{ g.csp_nonce }}"> // Código inline seguro con nonce </script>
2. ✅ Rate Limiting Implementado en Endpoints de Autenticación
✅ CORREGIDO
Categoría OWASP A07:2021 - Authentication Failures
Control ISO 27001 A.9.4.2 - Procedimientos seguros de inicio de sesión
Archivos Afectados app.py (líneas 588, 593, 606)

📋 Descripción

✅ IMPLEMENTADO (20/04/2026): Se ha implementado Flask-Limiter con límites específicos en todos los endpoints de autenticación. Los endpoints /login y /google-oauth tienen un límite de 10 requests por minuto, mientras que /authorize tiene un límite más restrictivo de 5 requests por minuto. Esto previene efectivamente ataques de fuerza bruta y enumeración de usuarios.

ISO/IEC 27001:2022 - Control A.9.4.2: Los procedimientos de inicio de sesión deben proteger contra intentos no autorizados mediante limitación de intentos fallidos y demoras incrementales.

✅ Código Implementado

# Configuración de Rate Limiting (app.py líneas 19-26) from flask_limiter import Limiter from flask_limiter.util import get_remote_address limiter = Limiter( app=app, key_func=get_remote_address, default_limits=["200 per day", "50 per hour"], storage_uri=os.getenv('REDIS_URL', 'memory://'), # ✅ Redis o memoria strategy='fixed-window' ) # Endpoints protegidos (app.py líneas 588-607) @app.route('/login') @limiter.limit("10 per minute") # ✅ Máximo 10 accesos/minuto/IP def login(): return render_template('login.html') @app.route('/google-oauth') @limiter.limit("10 per minute") # ✅ Protección OAuth def google_oauth(): redirect_uri = url_for('authorize', _external=True) return google.authorize_redirect(redirect_uri) @app.route('/authorize') @limiter.limit("5 per minute") # ✅ Más restrictivo para callback def authorize(): token = google.authorize_access_token() user_info = token.get('userinfo') # Error handler personalizado (app.py líneas 2472-2489) @app.errorhandler(429) def ratelimit_handler(e): logger.warning(f"Rate limit excedido - IP: {request.remote_addr}") return render_template('admin/error_429.html'), 429

🎯 Beneficios de la Implementación

Mejoras de seguridad logradas:

  • ✅ Protección contra ataques de fuerza bruta en login
  • ✅ Prevención de enumeración de usuarios autorizados
  • ✅ Límites granulares por endpoint (10/min login, 5/min authorize)
  • ✅ Página de error 429 personalizada con contador regresivo
  • ✅ Logging de intentos excesivos con IP tracking
  • ✅ Cumplimiento con ISO/IEC 27001:2022 Control A.9.4.2
  • ✅ Tests automatizados incluidos (test_rate_limiting.py)

Código de referencia original:

# pip install Flask-Limiter from flask_limiter import Limiter from flask_limiter.util import get_remote_address limiter = Limiter( app=app, key_func=get_remote_address, default_limits=["200 per day", "50 per hour"], storage_uri=os.getenv('REDIS_URL', 'memory://') ) @app.route('/login') @limiter.limit("10 per minute") # ✅ Máximo 10 accesos/minuto/IP def login(): return render_template('login.html') @app.route('/authorize') @limiter.limit("5 per minute") # ✅ Más restrictivo para OAuth callback def authorize(): token = google.authorize_access_token() user_info = token.get('userinfo')
3. Falta de Tokens CSRF Explícitos
Media
Categoría OWASP A01:2021 - Broken Access Control (CSRF)
Control ISO 27001 A.14.2.1 - Política de desarrollo seguro
Archivos Afectados app.py, templates/*.html (líneas 344, 409, 482, 1633, 1892)

📋 Descripción

La aplicación no implementa tokens CSRF explícitos. Aunque SESSION_COOKIE_SAMESITE='Lax' proporciona protección parcial, no es suficiente para navegadores antiguos o ciertos escenarios de navegación top-level. Flask no incluye CSRF protection por defecto, a diferencia de Django.

ISO/IEC 27001:2022 - Control A.14.2.1: La política de desarrollo seguro debe incluir controles establecidos como tokens anti-CSRF para operaciones que cambian estado.

💻 Código Actual

@app.route('/admin/users/add', methods=['GET', 'POST']) @require_admin_full def admin_add_user(): if request.method == 'POST': # ⚠️ Sin validación CSRF email = request.form.get('email') role = request.form.get('role') permissions_manager.add_user(email, role) # Template sin token CSRF <form method="POST"> <input name="email"> </form>

✅ Remediación Recomendada

Implementar Flask-WTF con protección CSRF global:

# pip install Flask-WTF from flask_wtf.csrf import CSRFProtect csrf = CSRFProtect(app) # ✅ Protección global automática @app.route('/admin/users/add', methods=['GET', 'POST']) @require_admin_full def admin_add_user(): if request.method == 'POST': # Flask-WTF valida CSRF automáticamente email = request.form.get('email') role = request.form.get('role') permissions_manager.add_user(email, role) # Template con token CSRF <form method="POST"> {{ csrf_token() }} <!-- ✅ Token generado automáticamente --> <input name="email"> </form>
4. Session ID Regeneration No Explícita
Baja
Categoría OWASP A07:2021 - Authentication Failures
Control ISO 27001 A.9.4.2 - Procedimientos seguros de inicio de sesión
Archivo Afectado app.py (líneas 664-672)

📋 Descripción

El logout usa session.clear() que limpia datos pero no regenera explícitamente el Session ID. Flask regenera automáticamente en ciertos casos, pero no está explícito en el código, lo que puede causar confusión o regresiones.

💻 Código Actual

@app.route('/logout') def logout(): session.clear() # ⚠️ No regenera Session ID explícitamente flash('Sesión cerrada exitosamente', 'success') return redirect(url_for('login'))

✅ Remediación Recomendada

@app.route('/logout') def logout(): username = session.get('username') session.clear() session.modified = True # ✅ Forzar regeneración de Session ID if username: logger.info(f"Usuario {username} cerró sesión") response = make_response(redirect(url_for('login'))) response.headers['Cache-Control'] = 'no-cache, no-store' flash('Sesión cerrada exitosamente', 'success') return response
5. subprocess sin shell=False Explícito
Baja
Categoría OWASP A03:2021 - Injection
Control ISO 27001 A.14.2.1 - Desarrollo seguro
Archivo Afectado src/security_audit.py

📋 Descripción

El uso de subprocess.run() no especifica explícitamente shell=False. Aunque es el default, la falta de explicitud puede causar confusión en code reviews o regresiones futuras. Esta es una mejora de hardening y documentación, no una vulnerabilidad activa.

💻 Código Actual

result = subprocess.run( ['bandit', '-r', project_path, '-f', 'json'], capture_output=True, text=True, timeout=timeout # ⚠️ shell=False es default pero no está explícito )

✅ Remediación Recomendada

import os import shlex def run_bandit_scan(project_path: str, timeout: int = 300): """ Ejecuta escaneo Bandit de forma segura. SEGURIDAD: Usa shell=False explícito y lista de argumentos. """ # Validar que project_path existe if not os.path.isdir(project_path): raise ValueError(f"Ruta inválida: {project_path}") command = ['bandit', '-r', project_path, '-f', 'json'] result = subprocess.run( command, capture_output=True, text=True, timeout=timeout, shell=False # ✅ Explícito: previene command injection ) return result
