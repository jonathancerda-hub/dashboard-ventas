Introducción al Dashboard de Ventas

Este manual describe el funcionamiento del Dashboard de Ventas, explicando cómo se obtienen, procesan y visualizan los datos desde el sistema Odoo. El objetivo es proporcionar una herramienta clara y precisa para el seguimiento del avance de ventas.

1. Obtención de Datos (desde Odoo)

La aplicación se conecta a Odoo para consultar las líneas de asientos contables (account.move.line). Estos datos iniciales se enriquecen con información de otros modelos relacionados para obtener una vista completa de cada transacción.

  • Asientos Contables (account.move): Fecha, estado, vendedor, etc.
  • Productos (product.product): Nombre, línea comercial, ciclo de vida, etc.
  • Clientes (res.partner): Nombre, NIF/RUC.
  • Órdenes de Venta (sale.order): Origen, observaciones de entrega.

2. Filtros Aplicados en la Consulta a Odoo

Para asegurar que solo se procesen datos relevantes, se aplican los siguientes filtros directamente en la consulta a la base de datos de Odoo:

Tipo de Movimiento: Solo facturas de venta y notas de crédito (move_id.move_type in ['out_invoice', 'out_refund']).

Estado del Movimiento: Solo asientos contables ya publicados (move_id.state = 'posted'). Las facturas en borrador ('draft') o canceladas ('cancel') NO se incluyen en ningún cálculo.

Código de Producto: Se excluyen productos que no tienen una referencia interna (product_id.default_code != False).

Categorías Excluidas: Se omiten productos de las siguientes categorías de servicio o internas:

  • MAT - ENVASE MEDIATO (ID: 315)
  • SERV_LOGISTICOS (ID: 333)
  • ART - MERCHANDISING (ID: 304)
  • MAT - ENVASE INMEDIATO (ID: 314)
  • MPF - EXCIPIENTE (ID: 318)
  • PCV - MARKETING (ID: 339)

Impuestos: Solo se incluyen líneas que tengan aplicados los impuestos "IGV" o "IGV_INC".

Importante sobre Estados: El sistema trabaja exclusivamente con facturas en estado 'posted'. Esto garantiza que solo se contabilicen transacciones confirmadas y publicadas en el sistema contable. Las facturas en borrador o canceladas no afectan ninguna métrica del dashboard.

3. Procesamiento y Lógica de Negocio (en la Aplicación)

Una vez que los datos son obtenidos de Odoo, la aplicación Flask aplica una serie de reglas de negocio para prepararlos antes de mostrarlos en el dashboard.

3.1 Inversión del Saldo (balance)

El campo balance de Odoo se multiplica por -1. Esto es crucial para la correcta interpretación de los datos:

  • Las ventas, que en contabilidad son un saldo negativo, se convierten en positivas.
  • Las notas de crédito, que son un saldo positivo, se convierten en negativas.

Importante: Gracias a este paso, el resto de la aplicación trabaja con números intuitivos y no necesita aplicar la función abs().

3.2 Exclusión de Ventas Internacionales

Se descartan todas las líneas de venta que pertenecen a operaciones internacionales para enfocar el análisis en el mercado local. El filtro se aplica si:

  • La línea comercial (commercial_line_national_id) contiene "VENTA INTERNACIONAL".
  • El canal de venta (sales_channel_id) contiene "VENTA INTERNACIONAL" o "INTERNACIONAL".

3.3 Agrupación de Datos y Cálculos de Métricas

Finalmente, los datos ya filtrados y procesados se agrupan para alimentar los diferentes componentes visuales del dashboard. Los cálculos más importantes son:

  • Venta Total: Suma del balance × (-1) de todas las ventas filtradas. Solo incluye transacciones en estado 'posted'.
  • Venta IPN (Productos Nuevos): Suma del balance × (-1) de las ventas de productos cuyo "Ciclo de vida del producto" (product_life_cycle) es exactamente 'nuevo'.
  • Vencimiento < 6 Meses: Suma del balance × (-1) de las ventas cuya "Ruta" (route_id) en la línea de pedido tiene el ID 18 ('3 > P < 6 meses') o 19 ('1 ≥ P ≤ 3 meses').
  • Ventas por Línea Comercial: Agrupación por el campo commercial_line_national_id del producto. Las líneas "GENVET" y "MARCA BLANCA" se normalizan automáticamente a "TERCEROS".
  • Ventas por Producto: Agrupación por nombre de producto (name). Los productos ATREVIA se consolidan eliminando indicadores de tamaño (MEDIUM, LARGE, SMALL, etc.).
  • Ventas por Ciclo de Vida: Agrupación por el campo product_life_cycle para el gráfico de dona.

Nota sobre Notas de Crédito: Las notas de crédito, al ser invertidas, aparecen con valores negativos en los cálculos. Esto significa que reducen automáticamente las ventas totales, reflejando correctamente las devoluciones o ajustes comerciales.

4. Uso del Dashboard

El dashboard principal es la primera página que verás después de iniciar sesión. Permite visualizar el avance de ventas de un mes específico. Puedes cambiar de mes usando el selector en la parte superior y los KPIs y gráficos se actualizarán automáticamente.

  • Personalización: El nombre del usuario que ha iniciado sesión se muestra debajo del título principal para una experiencia más personal.
  • Navegación: Desde aquí puedes acceder a las demás secciones como "Ventas", "Metas por Línea" y "Metas por Vendedor".
  • Exportación: Puedes exportar el detalle completo de los datos del mes seleccionado a un archivo Excel.

4.1 Tarjetas de KPIs (Indicadores Clave)

En la parte superior del dashboard se encuentran las tarjetas de KPIs que muestran métricas clave del desempeño comercial:

Fila Principal de KPIs:

  • Meta: Meta total de ventas establecida para el mes (suma de todas las líneas comerciales).
  • Venta: Ventas totales realizadas hasta el día actual (suma del campo balance invertido de Odoo).
  • % Avance: Porcentaje de cumplimiento de la meta total. Cálculo: (Venta Total / Meta Total) × 100
  • Meta IPN: Meta total de productos IPN (Introducción de Productos Nuevos) para el mes.
  • Venta IPN: Ventas de productos cuyo "Ciclo de vida del producto" en Odoo es exactamente 'nuevo'.
  • % Avance IPN: Porcentaje de cumplimiento de la meta IPN. Cálculo: (Venta IPN / Meta IPN) × 100

Fila de Avance Diario:

  • Avance Diario Acumulado: Proyección del avance si se mantiene el ritmo actual. Cálculo: [(Venta Total / Día Actual) × Días del Mes / Meta Total] × 100
  • Brecha Comercial (S/): Diferencia entre la meta y la venta actual. Cálculo: max(Meta Total - Venta Total, 0)
  • Avance Diario IPN Acumulado: Proyección del avance IPN con el ritmo actual. Cálculo: [(Venta IPN / Día Actual) × Días del Mes / Meta IPN] × 100
  • Brecha Comercial IPN (S/): Diferencia entre la meta IPN y la venta IPN actual. Cálculo: max(Meta IPN - Venta IPN, 0)

Estados de Odoo Utilizados: Todas las mediciones se realizan únicamente sobre asientos contables que están en estado 'posted' (publicados). Las facturas en borrador ('draft') no se incluyen en ningún cálculo.

4.2 Tabla Principal de Líneas Comerciales

La tabla central muestra el detalle de cada línea comercial con las siguientes columnas:

  • Línea Comercial: Nombre de la línea comercial (se excluyen LICITACION, NINGUNO, ECOMMERCE, GENVET y MARCA BLANCA de la visualización).
  • Meta: Objetivo de ventas para esa línea en el mes.
  • Venta: Ventas reales de la línea (suma del balance invertido).
  • % Cump: Porcentaje de cumplimiento. Cálculo: (Venta / Meta) × 100
  • % Total Vta.: Participación de la línea sobre el total de ventas. Cálculo: (Venta Línea / Total Ventas) × 100
  • Meta IPN: Meta de productos nuevos para esta línea.
  • Venta IPN: Ventas de productos con ciclo de vida 'nuevo'.
  • % Cump IPN: Porcentaje de cumplimiento IPN. Cálculo: (Venta IPN / Meta IPN) × 100
  • Venc <6 mss: Suma de ventas de productos cuya "Ruta" tiene ID 18 ('3 > P < 6 meses') o ID 19 ('1 ≥ P ≤ 3 meses').

Nota de Agrupación: Los productos de las líneas "GENVET" y "MARCA BLANCA" se agrupan automáticamente bajo "TERCEROS" para la visualización y los cálculos.

4.3 Gráficos Interactivos y Análisis

Top 7 Productos Más Vendidos

Este gráfico de barras horizontales muestra los 7 productos con mayores ventas en el periodo seleccionado. Los productos ATREVIA se agrupan automáticamente eliminando los indicadores de tamaño (MEDIUM, LARGE, SMALL) para consolidar las ventas por producto base.

  • Eje X: Monto de ventas en soles.
  • Eje Y: Nombre del producto (limpiado y consolidado).
  • Colores: Las barras se colorean según el ciclo de vida del producto (nuevo, consolidado, etc.).
  • Interactividad: Al pasar el cursor sobre una barra, se muestra el monto exacto de ventas.

Venta por Línea Comercial

Gráfico de barras verticales que muestra las ventas totales de cada línea comercial del mes.

  • Eje X: Líneas comerciales.
  • Eje Y: Monto de ventas en soles.
  • Fuente de Datos: Suma del balance invertido agrupado por commercial_line_national_id.

Venta por Tipo de Producto (Gráfico Donut)

Gráfico de dona que muestra la distribución de ventas según el ciclo de vida del producto.

  • Segmentos: Nuevo, Consolidado, Madurez, Declive, No definido.
  • Porcentajes: Cada segmento muestra su participación sobre el total de ventas.
  • Campo de Odoo: product_life_cycle del producto.
  • Interactividad: Al hacer clic en un segmento, se puede aislar para ver su valor específico.

4.4 Tabla Equipo ECOMMERCE

Si existe un equipo ECOMMERCE configurado en Google Sheets, se mostrará una tabla adicional con el desglose de sus ventas por línea comercial.

  • Filtro de Vendedores: Solo se incluyen ventas de los vendedores asignados al equipo ECOMMERCE.
  • Agrupación: Las ventas se agrupan por línea comercial.
  • Meta del Equipo: Se toma de las "Metas por Línea" configuradas en Google Sheets para la línea "ECOMMERCE".
  • % Avance: Se calcula comparando la venta total del equipo contra su meta asignada.

4.5 Gráfico de Ventas por Clasificación Comercial (DESACTIVADO)

Nota: Este gráfico está actualmente desactivado en el código pero puede reactivarse. Permite analizar la composición de las ventas de cada línea comercial según diferentes criterios.

  • Eje Vertical (Y): Muestra las principales líneas comerciales (Petmedica, Agrovet, etc.).
  • Eje Horizontal (X): Representa el total de ventas.
  • Barras Apiladas: Cada barra de una línea comercial está segmentada por colores, donde cada color representa una subcategoría.

Dimensiones Disponibles del Selector:

  • Forma Farmacéutica: Campo pharmaceutical_forms_id del producto.
  • Clasificación Farmacológica: Campo pharmacological_classification_id del producto.
  • Vía de Administración: Campo administration_way_id del producto.
  • Categoría de Producto: Campo categ_id del producto.
  • Línea de Producción: Campo production_line_id del producto.

Al seleccionar una nueva dimensión, el gráfico se redibuja automáticamente para reflejar la nueva estructura de datos, permitiéndote un análisis flexible y detallado de la composición de las ventas.

4.6 Resumen de Campos de Odoo Utilizados

Esta tabla resume los principales campos de Odoo que se utilizan en los cálculos y mediciones del dashboard:

Campos del Asiento Contable (account.move.line):

  • balance: Valor contable de la línea (se invierte: balance × -1)
  • move_id.state: Estado del asiento ('posted' = publicado, 'draft' = borrador, 'cancel' = cancelado)
  • move_id.move_type: Tipo de documento ('out_invoice' = factura, 'out_refund' = nota de crédito)
  • move_id.invoice_date: Fecha de la factura
  • partner_id: Cliente asociado

Campos del Producto (product.product):

  • name: Nombre del producto
  • default_code: Código o referencia interna del producto
  • commercial_line_national_id: Línea comercial nacional (PETMEDICA, AGROVET, TERCEROS, etc.)
  • product_life_cycle: Ciclo de vida ('nuevo', 'consolidado', 'madurez', 'declive')
  • categ_id: Categoría del producto
  • pharmaceutical_forms_id: Forma farmacéutica (tabletas, inyectable, etc.)
  • pharmacological_classification_id: Clasificación farmacológica
  • administration_way_id: Vía de administración (oral, tópica, etc.)
  • production_line_id: Línea de producción

Campos de la Orden de Venta (sale.order.line):

  • route_id: Ruta logística. Los IDs 18 y 19 identifican productos con vencimiento < 6 meses
  • sales_channel_id: Canal de ventas (se usa para filtrar ventas internacionales)

Importante: Todos los campos relacionales (terminados en _id) en Odoo devuelven una lista [ID, Nombre]. Por ejemplo: [875, 'PETMEDICA']. El sistema extrae el nombre (posición 1) para las agrupaciones y visualizaciones.

5. Gestión de Metas

La aplicación centraliza toda la gestión de metas y equipos en una única hoja de cálculo de Google Sheets, la cual se administra a través de dos interfaces en la aplicación web. Esto permite una gestión de datos centralizada y colaborativa.

Metas por Línea

Esta sección te permite establecer los objetivos de venta generales para cada línea comercial para un mes específico. Estos valores son los que se usan en el dashboard principal para calcular los porcentajes de avance globales. Los datos se guardan en la pestaña MetasPorLinea de tu Google Sheet.

Metas Vendedor (Gestión de Metas y Equipos)

Esta es la sección principal para la gestión detallada. Aquí puedes organizar a tus vendedores en equipos y asignarles metas individuales para cada mes del año en una vista de tabla dinámica. Los datos se guardan en las pestañas Equipos y Metas de tu Google Sheet, donde cada vendedor asignado a un equipo tiene su propia fila para mayor claridad.

  • Asignar Vendedores: Usa el selector "Asignar Vendedores" para añadir o quitar miembros de un equipo. Luego, presiona el botón "Actualizar Miembros". Esta acción guarda la nueva lista de integrantes en Google Sheets y recarga la página para que puedas ver las filas de los nuevos vendedores y asignarles sus metas.
  • Ingresar Metas: En la tabla, puedes ingresar la "Meta" total y la meta "IPN" para cada vendedor en la celda correspondiente a cada mes. Los campos numéricos aceptan separadores de miles para facilitar la lectura.
  • Guardar Cambios: Una vez que hayas ingresado todas las metas, presiona el botón "Guardar Todas las Metas y Equipos" al final de la página para almacenar todos los cambios en Google Sheets.

Importante: Toda la información de metas y equipos ahora reside en tu Google Sheet, eliminando la necesidad de archivos .json locales para esta tarea.

6. Dashboard por Vendedor

Al hacer clic en una línea comercial en la tabla del dashboard principal, se accede a una vista detallada centrada en el rendimiento de los vendedores para esa línea específica.

  • Filtros Dinámicos: Puedes cambiar de mes y de línea comercial directamente desde esta vista.
  • Tabla de Avance por Vendedor: Muestra un desglose del rendimiento de cada vendedor, incluyendo su meta, venta, avance IPN y la columna de "Vencimiento < 6 Meses".
  • Gráficos Comparativos: Incluye un gráfico de barras "Meta vs. Venta por Vendedor" para un análisis visual rápido del cumplimiento de objetivos.

7. Vista de Líneas de Venta

La página de "Ventas" ha sido optimizada para ser más rápida y fácil de usar.

  • Carga Rápida: La página ya no carga datos al entrar. La tabla aparecerá vacía inicialmente, garantizando una carga instantánea.
  • Búsqueda a Demanda: Los datos se obtienen únicamente cuando presionas el botón "Buscar".
  • Búsqueda sin Filtros: Si haces clic en "Buscar" sin especificar un rango de fechas, la tabla mostrará automáticamente los datos de los últimos 30 días.
  • Filtros Simplificados: Se ha eliminado el filtro por cliente para simplificar la interfaz y centrarse en los filtros más importantes (fechas y línea comercial).

8. Manejo de Errores y Seguridad

El Dashboard de Ventas implementa un sistema robusto de manejo de errores que cumple con los estándares ISO 27001 (Sección 5: Manejo de Errores). Este sistema protege la información sensible del sistema mientras proporciona una experiencia de usuario profesional y mantiene registros detallados para el equipo de desarrollo.

8.1 Manejadores Globales de Errores (@app.errorhandler)

La aplicación incluye 4 manejadores de errores globales que interceptan todos los errores antes de que lleguen al usuario:

1. Error 403 - Acceso Prohibido (@app.errorhandler(403))

  • Propósito: Se activa cuando un usuario intenta acceder a un recurso sin los permisos necesarios
  • Información Registrada: Email del usuario, ruta solicitada, timestamp
  • Mensaje al Usuario: Página personalizada indicando que no tiene permisos para acceder a ese recurso
  • Plantilla: templates/admin/error_403.html

2. Error 404 - Página No Encontrada (@app.errorhandler(404))

  • Propósito: Se activa cuando se solicita una URL que no existe en la aplicación
  • Información Registrada: Ruta solicitada, método HTTP, timestamp
  • Mensaje al Usuario: Página amigable con sugerencias de navegación y enlaces útiles
  • Plantilla: templates/admin/error_404.html

3. Error 500 - Error Interno del Servidor (@app.errorhandler(500))

  • Propósito: Se activa cuando ocurre un error interno en el servidor (excepción no manejada)
  • Información Registrada: Stack trace completo con exc_info=True, contexto del error, timestamp
  • Mensaje al Usuario: Página genérica indicando que hubo un problema temporal
  • Modo DEBUG: En desarrollo muestra detalles técnicos; en producción solo mensaje genérico
  • Plantilla: templates/admin/error_500.html

4. Manejador Genérico de Excepciones (@app.errorhandler(Exception))

  • Propósito: Captura cualquier excepción no manejada que no haya sido interceptada por los otros handlers
  • Información Registrada: Tipo de excepción, mensaje de error completo, stack trace, contexto
  • Mensaje al Usuario: Mensaje genérico sin revelar detalles técnicos
  • Seguridad: Previene la exposición accidental de información del sistema

Orden de Prioridad: Flask evalúa los handlers en orden de especificidad. Los códigos HTTP específicos (403, 404, 500) se evalúan antes que el handler genérico de Exception.

8.2 Plantillas Profesionales de Error

Cada tipo de error cuenta con una plantilla HTML profesional y responsive que proporciona una experiencia de usuario consistente:

Plantilla 403 - Acceso Prohibido

  • Diseño: Icono de candado con animación de "shake" (sacudida)
  • Información Mostrada: Mensaje amigable, rol del usuario actual, enlaces de navegación
  • Acciones: Botón para volver al Dashboard, contacto con administrador
  • Estilo: Colores cálidos (naranjas) para indicar advertencia sin ser alarmante

Plantilla 404 - Página No Encontrada

  • Diseño: Icono de brújula con animación de "bounce" (rebote)
  • Información Mostrada: Mensaje explicativo, sugerencias de páginas populares
  • Acciones: Enlaces rápidos a Dashboard, Análisis, Reportes, Metas
  • Estilo: Colores azules para mantener la calma y guiar al usuario

Plantilla 500 - Error Interno

  • Diseño: Icono de herramientas con animación de "pulse" (pulso)
  • Información Mostrada: Mensaje de disculpa, indicación de que el problema se registró
  • Acciones: Botón para reintentar, información de contacto de soporte
  • Modo DEBUG: Panel adicional con detalles técnicos (solo en desarrollo)
  • Estilo: Colores rojos suaves para indicar error sin crear pánico

Responsive Design: Todas las plantillas se adaptan automáticamente a dispositivos móviles, tablets y escritorio, manteniendo una experiencia profesional en cualquier pantalla.

8.3 Ofuscación de Mensajes de Error

Para prevenir la exposición de información sensible del sistema, la aplicación implementa una estricta separación entre los mensajes técnicos (para desarrolladores) y los mensajes genéricos (para usuarios):

Patrón de Implementación

ANTES (Inseguro):

  • Mensaje directo al usuario: flash(f'Error: {str(e)}', 'danger')
  • Problema: Expone detalles técnicos como rutas de archivos, queries SQL, nombres de variables
  • Riesgo: Puede revelar estructura del sistema a atacantes potenciales

DESPUÉS (Seguro):

  • Registro técnico: logger.error(f"Error al procesar datos: {e}", exc_info=True)
  • Mensaje al usuario: flash('Ocurrió un error al procesar tu solicitud. Por favor, inténtalo nuevamente.', 'danger')
  • Beneficio: Desarrolladores tienen acceso completo a logs, usuarios reciben mensaje útil sin detalles técnicos

Endpoints Actualizados (8 ubicaciones)

  • /verify_auth: Verificación de autenticación y tokens
  • /dashboard: Obtención de datos de ventas principales
  • /dashboard_linea/<linea>: Datos específicos por línea comercial
  • /metas: Procesamiento de metas y configuraciones
  • /export_data: Exportación de datos a Excel
  • /export_dashboard_detail: Exportación de detalles del dashboard
  • Y otros endpoints críticos

Logs Centralizados: Todos los errores se registran automáticamente en el directorio logs/ con rotación diaria, manteniendo un histórico completo para análisis y auditoría.
