1. Resumen Ejecutivo

El Dashboard de Ventas Farmacéuticas es una aplicación web diseñada para proporcionar visibilidad en tiempo real del desempeño de ventas de productos farmacéuticos a través de múltiples líneas comerciales. La plataforma integra datos de Odoo ERP con Supabase (PostgreSQL cloud) para ofrecer análisis detallados, seguimiento de metas, gestión de equipos de ventas y sistema completo de permisos RBAC.

🎯 Objetivo Principal: Centralizar y visualizar información crítica de ventas para facilitar la toma de decisiones estratégicas y operativas del equipo comercial.

2. Contexto del Negocio

2.1 Problema a Resolver

La empresa maneja múltiples líneas comerciales farmacéuticas (Petmedica, Agrovet, Pet Nutriscience, Avivet, Genvet, Interpet, entre otras) con equipos de vendedores distribuidos. Los desafíos principales incluyen:

  • Falta de visibilidad en tiempo real del avance de ventas vs. metas establecidas
  • Dificultad para analizar el rendimiento individual de vendedores por línea comercial
  • Datos dispersos entre diferentes sistemas (Odoo, hojas de cálculo)
  • Necesidad de identificar productos con próxima fecha de vencimiento
  • Seguimiento de productos nuevos (IPN - Introducción de Productos Nuevos)
  • Análisis del ciclo de vida de productos y formas farmacéuticas

2.2 Solución Propuesta

Una aplicación web centralizada que integre datos de ventas de Odoo ERP con Supabase PostgreSQL para gestión de metas, permisos y auditoría, proporcionando dashboards interactivos con KPIs, gráficos analíticos y capacidades de exportación para análisis detallado. Sistema completo de autenticación con Google OAuth2 y RBAC con 4 roles de acceso.

5.4 Integraciones

Odoo ERP (JSONRPC)

  • Extracción de líneas de venta con múltiples filtros
  • Obtención de lista de vendedores y equipos
  • Consulta de datos de productos, clientes, líneas comerciales
  • Datos de facturación y análisis de ventas

Supabase PostgreSQL (Cloud Database)

  • user_permissions: Usuarios autorizados con roles (admin_full, admin_export, analytics_viewer, user_basic)
  • audit_log_permissions: Registro de cambios en permisos (create, update, delete)
  • audit_log_login: Historial completo de login/logout con IP, User-Agent, session_id
  • audit_log_session_timeouts: Tracking de sesiones expiradas
  • analytics_page_views: Tracking de navegación por página
  • analytics_exports: Registro de exportaciones realizadas
  • analytics_sessions: Sesiones de usuario con duración y actividad

Google OAuth2 (Authlib)

  • Autenticación mediante cuentas Google corporativas (@agrovetmarket.com)
  • Verificación de dominio corporativo
  • Single Sign-On (SSO)
  • Tokens seguros con validación de scope

5.5 Seguridad

  • Autenticación: Google OAuth2 con Authlib 1.6.7, validación de dominio corporativo
  • RBAC (Role-Based Access Control): 4 roles implementados (admin_full, admin_export, analytics_viewer, user_basic)
  • Sistema de Permisos Granular: Tabla user_permissions en Supabase con is_active flag
  • Auditoría Completa: 4 tablas de audit_log (login, permissions, session_timeouts, general)
  • Session Security: 15min inactivity timeout + 8h absolute timeout, HttpOnly cookies, SameSite=Lax
  • Rate Limiting: Flask-Limiter con 200 requests/día y 50 requests/hora
  • Security Headers: CSP, X-Frame-Options, X-Content-Type-Options, HSTS implementados
  • Variables de Entorno: Credenciales en .env, no committed en Git
  • Manejo Global de Errores: 4 handlers (@app.errorhandler) que ofuscan información técnica
  • Plantillas de Error Profesionales: error_403.html, error_404.html, error_500.html con mensajes seguros
  • Logging Estructurado: Errores técnicos en logs con exc_info=True, mensajes genéricos a usuarios
  • Timezone Support: America/Lima (UTC-5) con pytz para timestamps consistentes
  • Cumplimiento ISO 27001: Sección 5 (91/100), preparación para A.9.4.2 (MFA en roadmap Q2 2026)

7. Reglas de Negocio

7.1 Cálculo de Métricas

Métrica Fórmula
% Avance (Venta Total / Meta Total) × 100
% Avance IPN (Venta IPN / Meta IPN) × 100
Avance Diario Acumulado (% Avance Total / Día Actual del Mes)
Ritmo Diario Requerido (100% - % Avance Actual) / Días Laborables Restantes
Proyección Lineal (Venta Actual / Día Actual) × Días Totales del Mes
Faltante para Meta max(Meta Total - Venta Total, 0)

7.2 Filtros y Exclusiones

  • Ventas Internacionales: Se excluyen automáticamente las ventas marcadas como "VENTA INTERNACIONAL" o del canal "INTERNACIONAL"
  • Días Laborables: Se consideran de Lunes a Sábado (excluyendo Domingo)
  • Productos con Vencimiento: Se identifican productos con route_id en [18, 19] (rutas de vencimiento <6 meses)
  • Productos Nuevos (IPN): Se consideran productos con product_life_cycle = 'nuevo'

7.3 Limpieza de Datos

🔧 Normalización de Productos ATREVIA: Los productos de la línea ATREVIA se agrupan eliminando los indicadores de tamaño/presentación (MEDIUM, LARGE, SMALL, etc.) para consolidar las ventas por producto base.

Ejemplos:

  • ATREVIA ONE MEDIUM → ATREVIA ONE
  • ATREVIA XR LARGE → ATREVIA XR
  • ATREVIA TRIO CATS SPOT ON MEDIUM → ATREVIA TRIO CATS

7.4 Gestión de Ajustes

  • Las Notas de Crédito sin vendedor asignado se agrupan en una fila especial "Ajustes y Notas de Crédito"
  • Los ajustes se suman al total de ventas de la línea para mantener consistencia contable
  • Los vendedores con ventas negativas (solo notas de crédito) se excluyen de la tabla, pero su impacto se refleja en el total

8. Despliegue y Configuración

8.1 Requisitos del Sistema

  • Python 3.11 o superior
  • Conexión a Odoo ERP (URL, base de datos, credenciales)
  • Cuenta Supabase con proyecto PostgreSQL configurado
  • Google OAuth2 credentials (Client ID + Client Secret)
  • Servidor web compatible con Gunicorn (Linux/Unix) o Render.com

8.2 Variables de Entorno (.env)

Variable Descripción
SECRET_KEY Clave secreta para sesiones de Flask
ODOO_URL URL del servidor Odoo
ODOO_DB Nombre de la base de datos Odoo
SUPABASE_URL URL del proyecto Supabase (ej: https://xxx.supabase.co)
SUPABASE_KEY Service Role Key de Supabase (anon key público)
GOOGLE_CLIENT_ID Client ID de Google OAuth2
GOOGLE_CLIENT_SECRET Client Secret de Google OAuth2

8.3 Archivos de Configuración

  • requirements.txt: Dependencias de Python (Flask 3.1.3, supabase 2.14.0, authlib 1.6.7, etc.)
  • .env: Variables de entorno (SUPABASE_URL, SUPABASE_KEY, GOOGLE_CLIENT_*, etc.)
  • .gitignore: Excluye .env, credentials.json, __pycache__

8.4 Estructura de Base de Datos Supabase

Tablas principales en PostgreSQL:

  • user_permissions: Columnas user_email | role | is_active | created_at | updated_at
  • audit_log_login: Columnas id | user_email | login_timestamp | ip_address | user_agent | session_id | oauth_provider
  • audit_log_permissions: Columnas id | action_type | admin_email | target_user_email | role_before | role_after | timestamp
  • analytics_page_views: Tracking de navegación con timestamp, user_email, page_url
  • analytics_exports: Registro de exports con formato, línea comercial, num_registros

Ver scripts SQL completos en: sql/create_permissions_tables_supabase.sql y sql/create_analytics_table_supabase.sql
